1. About NetPulse AI

NetPulse AI is a multi-agent telecom operations assistant. A natural-language complaint goes in — for example, "Major dropped calls in Surabaya since this morning" — and a structured incident ticket comes out, complete with related network events, CDR findings, and a recommended NOC action.

It was built for the APAC GenAI Academy 2026 Hackathon. The prototype ranked #82 in the Top 100 cut (announced 2026-04-23) and was refined against the Top 10 deadline of 2026-04-30.

Live demo. The Workspace page mocks a complete agent run end-to-end: the four sub-agents, the impact rollup, the saved ticket, and the recommended NOC action chips.

2. How it works

A single Sequential ADK agent runs four LlmAgent sub-agents in order:

  1. Classifier — extracts the issue category (billing, network, hardware, service, general) and the affected region from the raw complaint.
  2. Network investigator — queries BigQuery for recent events in the region, optionally rolling them up into a weekly trend via the weekly_outage_trend tool.
  3. CDR analyzer — poses one English question to AlloyDB AI; the database translates it into SQL, executes against the call-detail-records table under a read-only role, and returns rows.
  4. Response formatter — synthesises both findings into a single ticket, writes it to AlloyDB, and returns the assigned ticket_id.

Each agent picks its own model through a wrapper that walks a 3-attempt model ladder on quota or timeout failures (see §8). End-to-end runs target ≤15 s; current production traces sit around 10 s.

5. Schema reference

NetPulse is dataset-driven. The contract surface is three tables across two stores: one BigQuery table the network investigator reads, one AlloyDB table the CDR analyzer reads, and one AlloyDB table the response formatter writes.

5.5 Cross-table invariants

  1. Region vocabulary is shared across all three tables. The classifier extracts a region from the complaint; the investigator and analyzer filter by it; the formatter persists it. Mismatched vocabularies silently return zero rows and the ticket loses operational context.
  2. Category vocabulary is fixed by the agent contract. Five values only: billing · network · hardware · service · general.

6. Bring your own data

NetPulse will work against any dataset that matches the contract above. The bootstrap is a single shell script: 

export GOOGLE_CLOUD_PROJECT=your-project
export DATABASE_URL=postgresql+pg8000://postgres:<pwd>@<host>:5432/postgres
bash scripts/setup_byo.sh --seed

It runs two idempotent steps:

  1. scripts/setup_bigquery.py --recreate --seed — creates $BQ_DATASET.$BQ_NETWORK_TABLE with DAY-partition on started_at and clustering on (region, severity), then loads docs/seed-data/network_events.csv via WRITE_TRUNCATE.
  2. setup_alloydb.py --seed — creates $AL_CALL_TABLE and $AL_TICKET_TABLE, then loads docs/seed-data/call_records.csv and docs/seed-data/incident_tickets.csv using a single multi-row INSERT VALUES per file (one round-trip even at 5,000 rows).

6.1 Loading your own CSVs

Replace the three CSVs in docs/seed-data/ with your own files, keeping the column order from §5. Then re-run setup_byo.sh --seed. No code changes required as long as the column names and types match.

6.2 Renaming tables / datasets

Override the env vars in §5.1 before bootstrap. The agents and the data viewer tabs both consume the same env-driven names, so a single set of overrides retargets the entire stack. Example: 

export BQ_DATASET=acme_telco
export BQ_NETWORK_TABLE=outages_2026
export AL_CALL_TABLE=cdr_acme
export AL_TICKET_TABLE=tickets_acme
bash scripts/setup_byo.sh --seed
Vocabulary drift is the #1 cause of empty results. If your dataset uses different region names (e.g. province codes instead of city names), update the classifier prompt in telecom_ops/prompts.py so the extracted region matches your data. Without that step, the investigator and CDR analyzer will filter against the wrong strings and return zero rows.

7. AlloyDB AI NL2SQL

The CDR analyzer does not write hand-authored SQL. It calls query_cdr_nl(question) with one English question; the toolbox runs: 

SELECT alloydb_ai_nl.execute_nl_query(
  'netpulse_cdr_config',
  $1
) AS result;

The configuration netpulse_cdr_config is registered once via scripts/setup_alloydb_nl.py. That script is idempotent and walks ten steps:

  1. Install the alloydb_ai_nl extension
  2. Register a Vertex Gemini model via google_ml.create_model
  3. Create the NL config and bind the model
  4. Register public.call_records as a table view
  5. Add general-context strings (enums, region values, tower-ID format)
  6. Generate schema context (LLM call, 3-5 minutes, blocking)
  7. Apply the generated context
  8. Associate call_status with a concept type
  9. Build the value index
  10. Create the netpulse_nl_reader read-only role

The structural protection against destructive NL is the role — not the prompt. Even if the LLM emits a DROP TABLE, the role only has SELECT on call_records and EXECUTE on the alloydb_ai_nl helper functions, so the call errors out cleanly.

8. Vertex failover ladder

Every Vertex call goes through a small wrapper at telecom_ops/vertex_failover.py that walks a 3-attempt model ladder at the single global endpoint. Each attempt has its own timeout and an optional pre-sleep:

#ModelTimeoutPre-sleep
1primary (e.g. gemini-3.1-flash-lite-preview)10 s0 s
2primary (retry)20 s0.5 s
3gemini-2.5-flash (GA fallback)30 s0 s

Worst-case wall clock per agent: 60.5 s, well under Cloud Run's 300 s request timeout. The ladder swaps models, not regions, because preview models are gated to specific regions per project — region failover used to advance from global 429 to us-central1 404 NOT_FOUND every time.

10. Roadmap & future work

10.1 Custom schema support

The schema in §5 is opinionated to the Indonesian-telecom hackathon dataset. The next major step is to let an operator point NetPulse at their own schema without rewriting agent prompts:

  • Schema-aware classifier prompt. Today the classifier knows the 5 categories and 10 regions. Future: extract these from a YAML schema descriptor at boot, so a new deployment can declare its own category enum and region list.
  • Tool generation from schema descriptor. Generate the MCP Toolbox tools.yaml entries for query_network_events, query_cdr_nl, and save_incident_ticket from the same descriptor. One source of truth, no drift between prompts and SQL.
  • Pluggable analytical rollups. weekly_outage_trend assumes a started_at partition column. Future: declare partition + clustering keys in the schema descriptor and synthesise the rollup tool from those.
  • Automatic AlloyDB AI registration. Run setup_alloydb_nl.py automatically against whichever table is declared as the CDR source, including the general-context strings derived from the schema's enum columns.

10.2 Other directions on the queue

  • Vector search on historical tickets to surface the closest prior incident.
  • Multi-tenant routing (one deployment, many telcos, isolated AlloyDB databases).
  • Streaming responses through to the chat UI (currently stream=False to keep the failover ladder safe).
  • Replay mode: re-run a session against new prompts or models for regression testing.
  • Open-source the MCP Toolbox config + setup scripts as a reference template for other ADK + AlloyDB AI projects.
Want to contribute? NetPulse AI is built solo for the hackathon, but the schema descriptor + tool generation roadmap above is exactly the kind of refactor that benefits from outside eyes. Open an issue on the repo describing the data shape you want to point NetPulse at, and we'll prioritise the descriptor format around real cases.
